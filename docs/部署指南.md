# 部署指南

## 架构概述

Manila Web UI 是一个**纯前端应用**，所有逻辑（游戏引擎、AI 推理）都在浏览器端运行：

```
浏览器
├── React UI (Vite 构建)
├── @manila/engine (游戏引擎，TypeScript)
├── @manila/rl (状态编码 + 动作映射)
├── @manila/strategy (到港概率计算)
└── onnxruntime-web (加载 ONNX 模型做 AI 推理)
```

不需要后端服务器，可以部署到任何静态托管平台。

## 当前部署方式：GitHub Pages

- **仓库**：https://github.com/ntuht/Manila_webui
- **访问地址**：https://ntuht.github.io/Manila_webui/
- **自动部署**：每次 push 到 `main` 分支，GitHub Actions 自动构建并部署

### 更新流程

```bash
git add -A
git commit -m "修改描述"
git push
# ~2 分钟后自动部署生效
```

## 项目结构与 npm Workspaces

```
Manila_webui/
├── package.json          # 根目录，配置 workspaces
├── client/               # React 前端 (Vite)
├── packages/
│   ├── engine/           # 游戏引擎
│   ├── rl/               # RL 编码器 + 动作映射
│   └── strategy/         # 策略计算（到港概率等）
└── .github/workflows/    # CI/CD
```

根 `package.json` 配置了 workspaces：
```json
{
  "workspaces": ["packages/*", "client"]
}
```

这使得 `npm ci` 一次安装所有子包依赖，并自动建立包之间的 symlink。

## 部署踩坑记录

### 问题 1：`@manila/strategy` 模块找不到

**现象**：CI 构建报 `Cannot find module '@manila/strategy'`

**原因**：`client` 通过 Vite alias 引用 `@manila/rl`，而 `encoder.ts` import 了 `@manila/strategy`。本地因为 workspace symlink 能自动解析，但 Vite alias 只配了 engine 和 rl，缺少 strategy。

**修复**：在 `vite.config.ts` 添加 `@manila/strategy` 的 alias：
```ts
resolve: {
  alias: {
    '@manila/engine': path.resolve(__dirname, '../packages/engine/src/index.ts'),
    '@manila/rl': path.resolve(__dirname, '../packages/rl/src/index.ts'),
    '@manila/strategy': path.resolve(__dirname, '../packages/strategy/src/index.ts'),
  },
},
```

**教训**：Monorepo 中 A → B → C 的依赖链，每一层都必须在构建工具中显式配置。本地隐式解析不代表 CI 也行。

---

### 问题 2：TypeScript 类型检查在 CI 报错

**现象**：`tsc -b` 报大量类型错误（`TS2307`, `TS6133`）

**原因**：`client/package.json` 的 build 脚本是 `tsc -b && vite build`。`tsc` 做严格类型检查时无法通过 Vite alias 解析模块类型声明。本地 `tsc --noEmit` 能过是因为 `node_modules` 有完整的类型链。

**修复**：CI 的 build 步骤改为直接 `npx vite build`，跳过 `tsc`。Vite 内部用 esbuild 编译 TS，不做类型检查。

**教训**：
- `tsc` 和 Vite 的类型解析机制不同
- CI 部署只需要构建产物，类型安全在开发阶段通过 IDE 和 pre-commit hook 保证
- 可以把 `tsc --noEmit` 作为单独的 lint 步骤，而非阻塞构建

---

### 问题 3：`@vitejs/plugin-react` 包找不到

**现象**：`Could not find package '@vitejs/plugin-react'`

**原因**：CI workflow 的 `npm ci` 在项目根目录执行，但 `@vitejs/plugin-react` 是 `client/package.json` 的 devDependency。根目录没有配置 workspaces 包含 client，所以不会安装 client 的依赖。

**修复**：
1. 临时方案：在 CI 中增加 `npm ci --prefix client` 步骤
2. 根本方案：在根 `package.json` 的 `workspaces` 中加入 `"client"`

**教训**：npm workspaces 需要显式声明所有工作区，遗漏任何一个都会导致其依赖不被安装。

---

### 问题 4：修改 workspaces 后 lock file 不匹配

**现象**：CI 报 `Missing: autoprefixer@10.4.27 from lock file`

**原因**：在根 `package.json` 的 `workspaces` 中新增了 `"client"` 后，`package-lock.json` 还是旧的（不包含 client 的依赖树）。`npm ci` 严格要求 lock file 与 `package.json` 一致。

**修复**：本地运行 `npm install --package-lock-only` 重新生成 lock file，然后提交。

**教训**：**任何 workspaces 配置变更后必须重新生成 lock file**。`npm ci` 不像 `npm install` 会自动修复，它是严格匹配模式。

---

## 关键配置文件

### `vite.config.ts` — GitHub Pages base 路径

```ts
base: process.env.GITHUB_PAGES ? '/Manila_webui/' : '/',
```

通过环境变量控制，本地开发用 `/`，CI 部署用仓库名作为子路径。

### `.gitignore` — 排除的大文件

| 排除项 | 原因 |
|-------|------|
| `packages/rl/python/runs/` | 训练检查点、TensorBoard 日志（数百 MB） |
| `manila_replay_*.json` | 游戏复盘记录 |
| `node_modules/` | npm 自动安装 |
| `dist/`, `build/` | CI 自动构建 |
| `*.pyc`, `__pycache__/` | Python 缓存 |

### ONNX 模型

- 位置：`client/public/models/manila_brain.onnx`（~2MB）
- 随代码一起上传到 GitHub，构建时自动打包到 `dist/models/`
- 浏览器首次加载后会缓存

## 通用经验总结

1. **先本地模拟 CI 构建**：用干净环境（删除 node_modules 后重装）跑一次 `npm ci && npm run build`，提前发现依赖问题
2. **Monorepo 的依赖链要完整**：确保所有传递依赖在构建环境中可达
3. **分离类型检查和构建**：CI 部署时只做 `vite build`，类型检查留给开发
4. **npm workspaces 要完整配置**：所有子包都要在 `workspaces` 字段中声明
